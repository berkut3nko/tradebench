# Архітектурні та програмні патерни у проєкті TradeBench

Використання стандартизованих патернів проєктування забезпечило гнучкість, масштабованість та високу швидкість розробки платформи TradeBench. Нижче наведено опис ключових рішень на різних рівнях системи.

## 1. Архітектурні патерни (Architectural Patterns)

1.1. Мікросервісна архітектура (Microservices)

Система розділена на два незалежних вузли:

- Оркестратор (PHP): Керує життєвим циклом користувачів, API та інтеграціями.

- Обчислювальне ядро (C++): Ізольований сервіс для інтенсивних CPU-обчислень.

- Перевага: Це дозволяє масштабувати обчислювальні потужності незалежно від веб-сервера та використовувати найбільш підходящу мову програмування для кожної конкретної задачі.

1.2. MVC (Model-View-Controller)

Класичний патерн, реалізований у межах PHP-бекенду:

- Model: Класи CurrencyData, AnalysisTask — відповідають за взаємодію з PostgreSQL.

- View: HTML-шаблони landing.html, dashboard.html — інтерфейс користувача.

- Controller: AuthController, AnalysisController, AdminController — координують потік даних між моделлю та відображенням.

1.3. Front Controller

Реалізовано у файлі [public/index.php](/php-app/public/index.php). Усі запити проходять через єдину точку входу, де ініціалізується середовище, завантажуються налаштування .env та запускається маршрутизатор.

- Перевага: Централізована обробка безпеки (JWT), логування та маршрутизації.

## 2. Комунікаційні патерни (Communication Patterns)

2.1. Remote Procedure Call (gRPC)

Використовується для синхронного виклику функцій C++ ядра з боку PHP.

- Механізм: Контракт описується у файлі analysis.proto, а серіалізація Protobuf гарантує мінімальний розмір повідомлень.

2.2. Publish/Subscribe (Redis Pub/Sub)

Використовується для асинхронного зв'язку "C++ -> PHP".

- Механізм: Після завершення обчислень C++ публікує ID завдання в канал Redis. PHP-скрипт SSE, будучи "підписником", отримує це повідомлення та передає його у браузер клієнта.

## 3. Програмні патерни (Design Patterns GoF)

3.1. Singleton (Одинак)

Реалізовано в класі [App\Core\Database](/php-app/src/Core/Database.php).

- Опис: Гарантує, що протягом одного життєвого циклу PHP-запиту буде створено лише одне з'єднання з базою даних PostgreSQL.

- Код: Метод Database::getConnection() повертає існуючий екземпляр PDO, якщо він вже ініціалізований.

3.2. Strategy (Стратегія)

Реалізовано в обчислювальному ядрі C++ (клас [BacktestingEngine](/cpp-core/src/engine.cpp)).

- Опис: Система підтримує різні алгоритми бектестингу (SMA, RSI, MACD). Замість використання громіздких умов if/else у кожному місці, ядро обирає конкретний алгоритм виконання на основі типу стратегії в запиті.

3.3. Observer (Спостерігач)

Реалізовано через технологію Server-Sent Events (SSE) на [фронтенді](/php-app/public/assets/js/dashboard.js).

- Опис: Веб-інтерфейс "спостерігає" за потоком подій від сервера. Як тільки в потоці з'являються нові дані, інтерфейс автоматично реагує — оновлює графіки та віджети статистики без участі користувача.

3.4. Middleware (Посередник)

*Реалізовано в класі [AuthMiddleware](/php-app/Http/Middleware/AuthMiddleware.php).*

- Опис: Шар перевірки, який стоїть між маршрутизатором та контролером. Він перевіряє JWT токен та права доступу (RBAC) перед тим, як дозволити запиту дійти до бізнес-логіки.

## 4. Патерни роботи з даними (Data Patterns)

4.1. DAO / Repository (Об'єкт доступу до даних)

- Реалізовано як у PHP (класи в App\Models), так і в C++ (DataRepository).

- Опис: Логіка формування складних SQL-запитів винесена з контролерів у спеціалізовані класи.

- Перевага: Зміна структури таблиць у БД вимагає виправлення коду лише в одному місці (моделі), не зачіпаючи логіку API або математичні розрахунки.

4.2. Smart Caching

- Реалізовано в логіці завантаження даних з Binance.

- Опис: Перед кожним запитом до зовнішнього API система перевіряє наявність даних у локальній PostgreSQL. API-виклик здійснюється лише для "дірок" у даних.
