# Архітектура TradeBench: Від класичного MVC до розподілених систем

## 1. Розподілена архітектура (Distributed MVC)

Наша система не є класичним монолітом. Ми використовуємо розподілену архітектуру, де роль "Моделі" та "Контролера" розділена між різними технологіями.

### Розподіл компонентів:

**Controller (Оркестратор) — PHP**
PHP у нашій системі виконує роль "головного мозку" для веб-інтерфейсу:

* Приймає HTTP-запити.
* Перевіряє JWT (авторизація).
* **Оркестрація:** Замість того, щоб самому рахувати складну математику, він через gRPC дає команду C++ серверу.
* Повертає клієнту відповідь "Завдання прийнято".

**Model (Обчислювальне ядро) — C++ + PostgreSQL**
У нашому випадку Модель розділена на дві частини:

* **Data Layer (PostgreSQL):** Сховище "сирих" даних та результатів.
* **Logic Layer (C++ Core):** Виконує роль "важкої" моделі. Він робить те, що PHP робив би надто повільно: паралельну обробку мільйонів тіків та розрахунок індикаторів.

**View (Інтерфейс) — JavaScript/HTML**

* Відображає дані, які PHP дістає з бази після завершення роботи C++.
* Отримує сигнали через WebSockets (Real-time View) про зміну стану завдань.

## 2. Programming Principles (Принципи програмування)

Під час розробки programmatic-комплексу суворо дотримувалися фундаментальні принципи інженерії програмного забезпечення для забезпечення чистоти, масштабованості та підтримуваності коду.

1. **Separation of Concerns (SoC) — Розділення відповідальності:**
Система фізично та логічно розділена на веб-оркестратор (PHP) та обчислювальне ядро (C++). Жоден із цих мікросервісів не знає про внутрішню реалізацію іншого, вони спілкуються виключно через контракти Protobuf.
2. **Single Responsibility Principle (SRP) — Принцип єдиної відповідальності (із SOLID):**
Кожен клас у системі має лише одну причину для змін. Наприклад, клас [BinanceService.php](php-app/src/Services/BinanceService.php) відповідає виключно за HTTP-запити до зовнішнього API біржі, а клас [CurrencyData.php](php-app/src/Models/CurrencyData.php) — виключно за збереження цих даних у базу PostgreSQL. Вони не змішуються.
3. **KISS (Keep It Simple, Stupid) — Зберігай простоту:**
PHP-бекенд був написаний без використання "важких" фреймворків (таких як Laravel чи Symfony). Замість цього розроблено власне легкотривале ядро маршрутизації ([Router.php](php-app/src/Core/Router.php)), що дозволило максимізувати продуктивність та зробити логіку максимально прозорою.
4. **DRY (Don't Repeat Yourself) — Не повторюйся:**
Логіка форматування HTTP-відповідей (успіх/помилка) винесена в окремий статичний клас [Response.php](php-app/src/Core/Response.php). Це дозволило уникнути дублювання коду встановлення HTTP-заголовків (`header('Content-Type: application/json')`) та `json_encode` у кожному контролері.
5. **Dependency Inversion Principle (DIP) — Принцип інверсії залежностей (із SOLID):**
Високорівневий модуль (PHP) та низькорівневий модуль (C++) не залежать один від одного напряму. Обидва модулі залежать від єдиної абстракції — інтерфейсу AnalysisService, який строго визначений у файлі [analysis.proto](proto/analysis.proto).

## 3. Design Patterns (Патерни проєктування)

Для вирішення типових архітектурних завдань у проєкті було використано стандартизовані патерни проєктування (GoF та архітектурні патерни).

### 3.1. Front Controller (Єдина точка входу)

* **Файл:** [index.php](php-app/public/index.php)
* **Пояснення:** Усі HTTP-запити до бекенду спрямовуються веб-сервером Nginx до одного єдиного файлу `index.php`. Цей патерн використано для централізованої ініціалізації середовища (завантаження .env), налаштування CORS-заголовків, глобального перехоплення помилок та делегування запитів до MVC-маршрутизатора. Це унеможливлює обхід систем безпеки.

### 3.2. Singleton (Одинак)

* **Файл:** [Database.php](php-app/src/Core/Database.php)
* **Пояснення:** Використано для управління з'єднанням з PostgreSQL (PDO). Патерн гарантує, що протягом одного життєвого циклу HTTP-запиту буде створено лише один екземпляр з'єднання з базою даних. Це суттєво економіть пам'ять та запобігає вичерпанню ліміту пулу з'єднань (Connection Pool) під навантаженням.

### 3.3. Strategy (Стратегія)

* **Файл:** [engine.cpp](cpp-core/src/engine.cpp) (див. метод runSimulation)
* **Пояснення:** Використано в обчислювальному ядрі C++. Дозволяє системі динамічно обирати та виконувати різні алгоритми бектестингу (SMA, MACD, RSI, Bollinger Bands) під час виконання програми на основі рядка strategy_payload, отриманого через gRPC. Це дозволяє додавати нові торгові стратегії без зміни основного циклу симуляції.

### 3.4. Publish-Subscribe (Видавець-Підписник / Спостерігач)

* **Файли:** [grpc_service.cpp](cpp-core/src/grpc_service.cpp) (Видавець) та [AnalysisController.php](php-app/src/Controllers/AnalysisController.php) (Підписник).
* **Пояснення:** Використано для асинхронного сповіщення клієнта. C++ ядро публікує подію в канал Redis після завершення обчислень. PHP-метод stream() (який утримує відкрите SSE-з'єднання з браузером) "спостерігає" за цим каналом і миттєво відправляє подію на фронтенд, оновлюючи графіки у реальному часі.

## 4. Refactoring Techniques (Техніки рефакторингу)

Протягом розробки проєкту код постійно покращувався для забезпечення його читабельності та легкості підтримки. Були використані наступні техніки рефакторингу:

1. **Extract Method (Відокремлення методу):**
Складні математичні розрахунки в C++ ядрі (наприклад, обчислення Експоненційного ковзного середнього чи Індексу відносної сили) були винесені з основного циклу runSimulation в окремі приватні хелпер-методи (calculateEMA, calculateRSI) у файлі [engine.cpp](cpp-core/src/engine.cpp). Це значно скоротило розмір головної функції та покращило її читабельність.
2. **Extract Class (Відокремлення класу / Виокремлення DAO):**
Спочатку логіка SQL-запитів знаходилася безпосередньо всередині контролерів. Під час рефакторингу вона була винесена в спеціалізовані Data Access Object (DAO) класи: [database.cpp](cpp-core/src/database.cpp) у C++ та [CurrencyData.php](php-app/src/Models/CurrencyData.php), [AnalysisTask.php](php-app/src/Models/AnalysisTask.php) у PHP.
3. **Replace Magic Numbers with Constants/Environment Variables (Заміна магічних чисел):**
Жорстко закодовані (hardcoded) доступи до баз даних, API-ключі Gemini та секретні ключі для підпису JWT токенів були вилучені з коду і замінені на звернення до змінних середовища (через бібліотеку phpdotenv).
4. **Move Method (Переміщення методу):**
Логіка взаємодії зі штучним інтелектом (Google Gemini) була вилучена з загального [AnalysisController.php](php-app/src/Controllers/AnalysisController.php) і переміщена до спеціально створених для цього [AiController.php](php-app/src/Controllers/AiController.php) та [AiService.php](php-app/src/Services/AiService.php), щоб не порушувати принцип єдиної відповідальності (SRP).
5. **Encapsulate Field (Інкапсуляція поля):**
Властивості класів (наприклад, екземпляр PDO $db у контролерах або AnalysisServiceClient $client) були переведені з модифікатора доступу public на private у відповідних файлах, наприклад у [AuthMiddleware.php](php-app/src/Core/AuthMiddleware.php). Доступ до них або їхня зміна тепер можливі виключно через конструктор, що захищає внутрішній стан об'єктів від непередбачуваних змін ззовні.

├── php-app/                # PHP gRPC Клієнт (Веб-бекенд)
│   ├── src/                # MVC Архітектура (Controllers, Models, Services)

│   ├── tests/Unit/         # PHPUnit тести
